整體版面

│icon│  清單側欄      │        主內容區           │  詳情/編輯面板  │
│rail│ (可摺疊)      │  (任務列表/月曆/看板/習慣)  │  (點項目才展開) │

左側 icon rail 固定不變（任務✓、月曆、目標🎯、番茄鐘⏱、看板▦、商店、搜尋），代表這是四大模組共用一個殼、切換視圖而已 — 資料很可能是共通的（同一個 Task 在任務列表、看板、月曆都會出現）。

四大模組拆解

1. 任務清單 (Tasks)
- 智慧清單：今天／最近7天／收集箱／已完成
- 自訂清單：TODO、LIFE、EXEOLY、Leisure、Wishlist、Futures（有顏色圓點+未完成數量），可再用「資料夾」(NOTES) 分組
- 每個任務：標題、到期日、優先級旗標(紅)、所屬清單、標籤(攝影/狗狗/運動項目)
- 「已過期」「習慣」自動分組區塊

2. 月曆 (Calendar)
- 月/週/日三種視圖切換
- 同一批任務直接畫在日期格子裡（顏色=清單顏色），可見「每日打卡」這種重複性任務每天都出現 → 代表 Task 本身有重複規則(recurrence)
- 點任務彈出跟任務列表一樣的編輯面板

3. 看板 (Kanban)
- 每個清單可以切換成看板視圖：TODO / In progress / Done 三欄
- 這其實只是「同一份任務資料」換一種分組方式（依狀態而非依日期）→ 看板欄位很可能對應 Task 的 status 欄位

4. 習慣 (Habits)
- 獨立模組但沿用同一個殼
- 週視圖打卡格 + 每個習慣一行，顯示「⚡連續天數 🔥累積天數」
- 詳情頁：月打卡數、月完成率、當前連續、月曆熱力圖式打卡紀錄

我推的資料模型雛形

List(清單)  1───* Task(任務) *───1 Status(todo/doing/done)
                    │
                    ├─ dueDate/dueTime, priority, tags[], recurrence
                    └─ 出現在: TaskList視圖 / Calendar視圖 / Kanban視圖 (同一份資料三種投影)

Habit(習慣)  1───* CheckinRecord(某天是否打卡)
   (獨立於 Task，但共用 icon/顏色系統)

Tag(標籤)  *───* Task

這樣設計的話，Task 是核心實體，Calendar 和 Kanban 都只是它的「視圖」，不是獨立資料源 — 這是我認為最關鍵的架構決策，值得先確認你是否認同。





# 完整規劃總結

技術棧：Next.js 全端（App Router）+ TypeScript + SQLite + Prisma + Auth.js + Server Actions

MVP 範圍：任務清單模組
- 登入/註冊
- 清單 CRUD（顏色、未完成數量、資料夾分組）
- 智慧清單：今天／最近7天／收集箱／已完成／已過期（手動順延）
- 任務 CRUD（標題、到期日、優先級旗標、備註、標籤）
- 標籤 CRUD + 篩選
- 不含：重複規則、月曆/看板/習慣視圖（Task schema 預留 dueDate/status 欄位相容）

RWD + PWA
- 桌機：icon rail + 清單側欄 + 主內容 + 右側詳情面板（四欄）
- 手機：單欄主內容 + 底部導航列，側欄改抽屜，詳情面板改全螢幕頁
- PWA：manifest + service worker 快取靜態資源與 App shell，任務資料操作仍需連網（Server Actions 本質上就需要網路）
